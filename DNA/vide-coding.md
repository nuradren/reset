## 1) Чтобы не утонуть в “vibe-based” хаосе: spec-first и фиксация решений

### GitHub Spec Kit (spec-first подход)

Это ключевая находка из лога, потому что решает вашу главную боль: «агенты быстро генерят, но потом всё переписывают и канон расползается». `github/spec-kit` прямо позиционируется как средство борьбы с “vibe-based programming” через **спеки на естественном языке до генерации**. 

Как применить к сериалу:

* Спека = **контракт серии/сцены** (цель, раскрытия тайны, что меняется в герое, какие “нити” продвинуть, какие нельзя трогать).
* После спеки агент пишет outline/сцену, а ревью‑агенты проверяют “тесты” (см. раздел про quality-gates).

Ссылка:

```text
https://github.com/github/spec-kit
```

### “vibe-coding-cn” как методология работы с агентами

В логе упоминается целый гайд/репо про “vibe coding” (в т.ч. с упором на системность). Это полезно не как «копировать 1:1», а как набор паттернов: правила, контуры ответственности, протоколы задач. 

Ссылка:

```text
https://github.com/tukuaiai/vibe-coding-cn
```

## 2) Параллельная работа нескольких агентов без конфликтов: worktree / PR‑поток / “фоновые” агенты

### Git worktree (и поддержка в Cursor)

В логе прямо есть тезис, что **Cursor умеет git worktree**, что идеально для вашей модели: «каждый агент работает в своей песочнице, не ломая main». 

Как применить:

* `worktree` = отдельная «комната сценаристов» под ветку `s01e03-outline`, `character-protagonist-refactor`, `lore-arg-protocol`, и т.д.
* Вы можете поручать агентам задачи параллельно (персонажи / нити / серия), а потом мерджить через PR.

### git-worktree-runner

В логе есть репозиторий CLI‑утилиты `coderabbitai/git-worktree-runner`, которая помогает управлять worktree. 

Ссылка:

```text
https://github.com/coderabbitai/git-worktree-runner
```

### Graphite для PR‑флоу

В логе Graphite позиционируется как более удобная система работы с PR (и важная деталь: “работать с агентом прямо внутри открытых PR”). 

Как применить:

* Ваш “writer’s room” становится похож на инженерный процесс: серия/нить = PR, а агенты комментируют, предлагают правки, вы принимаете/отклоняете.

Ссылка (как продукт):

```text
https://graphite.dev/
```

### Cursor 1.0: BugBot + Background Agent

В логе упомянуто, что у Cursor есть **BugBot (авто‑ревью PR)** и **Background Agent** (задачи “в фоне”), плюс расширенная память и более удобное подключение MCP. 

Как применить к сериалу (прямо по смыслу, даже если это “для кода”):

* BugBot‑логика = ваш **Continuity Guardian**: ловит противоречия, “мертвые нити”, несостыковки таймлайна.
* Background Agent = “ночная смена”: агент в фоне делает выжимки интервью, раскладывает по карточкам персонажей, предлагает варианты сцен.

## 3) Quality-gates против “слопа”, банальностей и расползания канона

### stop-slop — фильтр «ИИ‑сладкой воды»

В логе прямо упомянут репозиторий `hardikpandya/stop-slop` — по смыслу это инструмент “против слопа” (типовой AI‑стилистики). 

Как применить у вас:

* Не обязательно запускать как код‑линтер; даже как **принцип**: прогоняем новые сцены через “анти‑слоп” проверку, вычищаем клише, “обобщалки”, неестественные формулировки.
* Можно сделать шаг в CI: “если сцена содержит N стоп‑слов/паттернов — просим переписать”.

Ссылка:

```text
https://github.com/hardikpandya/stop-slop
```

### PR‑ревью “как сервис”: трюк с `0` в URL PR

В логе есть лайфхак: добавить `0` в URL Pull Request, чтобы ИИ помог понять изменения. 

Как применить:

* Для вас PR = “изменения канона”. Такой обзор помогает быстро понять “что именно поменялось” в персонаже/нити/серии, прежде чем мерджить.

### Авто‑ревью инструментами вокруг IDE (CodeRabbit и др.)

В логе упоминается CodeRabbit с AI‑ревью в IDE. 
Для вас это полезно как модель: **авто‑ревью текста** (не кода) по чек‑листу: логика, ясность, конфликт, эмоциональная динамика, канон.

### Практика: общий CLAUDE.md в репо, который команда дополняет

В логе описан паттерн: в репозитории лежит общий `CLAUDE.md`, который команда регулярно обновляет, когда замечает, что агент делает что-то не так.

# Как собрать из этого единую систему под ваш сериал+проект

Ниже — **рабочий “минимальный стек”**, который не перегрузит вас, но даст мультиагентность и контроль.

## Минимум (на старт)

1. **GitHub repo + Issues/Projects** (это база).
2. **Spec-first**: `github/spec-kit` как метод (не обязательно как утилита) — “сначала спека серии/нити, потом текст”. 
3. **AGENTS.md + CLAUDE.md** в корне репо — “конституция” комнаты сценаристов.
4. **Worktrees** для параллельной работы (Cursor / git-worktree-runner).
5. **stop-slop** как quality-gate для текста. 

## Расширение (когда появится 2–3 соавтора/агента и станет тяжело)

6. **Graphiti** как память‑граф канона (персонажи/нити/артефакты/таймлайн). 
7. **GitHub MCP Server + MCP Manager** для удобного агентного управления репо/задачами.
8. **repo-to-chatgpt / 1file** для ускоренной загрузки контекста в агента.

# Практическая “врезка” в ваш репо (что добавить, чтобы агенты могли работать)

Я бы сделал так (вы можете перенести в вашу текущую структуру, не ломая её):

* `CLAUDE.md` — глобальные правила (тон, стиль, что нельзя ломать в каноне). 
* `AGENTS.md` — роли и протокол работы (Showrunner/Interviewer/Continuity/Critic/Dialogue). И подключить его из `CLAUDE.md`. 
* `specs/` — спеки (в духе spec-kit), хотя бы:

  * `specs/episode_template.md`
  * `specs/character_template.md`
  * `specs/thread_template.md`
* `.github/ISSUE_TEMPLATE/`:

  * `episode-outline.yml`
  * `canon-change.yml`
  * `research.yml`
* `.github/pull_request_template.md`:

  * чек‑лист: “что изменено”, “какие нити продвинуты”, “какие раскрытия добавлены”, “что нельзя сломать”.
